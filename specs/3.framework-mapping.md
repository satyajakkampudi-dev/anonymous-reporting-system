# Framework Mapping — Anonymous Reporting System

> LoG.ai Layer 3. Maps each `rendering` value in the input schemas to exact frontm.ai framework
> primitives. **Every primitive here is documented in `docs/`** — composing documented primitives is
> NOT inventing; inventing is calling a method/property/event that appears in no doc. This is the
> bridge that prevents the code generator from silently downgrading `custom_card` to `standard`.
>
> Applies to **both** micro-apps (anonymous-user and anonymous-admin). Companion to
> [`3.field-spec.md`](3.field-spec.md), [`3.anonymous-user-input-schema.md`](3.anonymous-user-input-schema.md).

## Two-Doc architecture

Both apps use the **Two-Doc** pattern (custom HTML cards per REQUIREMENTS §9):

- **Data Doc** (`reportDoc`) — owns ALL sections with Field objects, the embedded sub-collections
  (`amendments`, `statusHistory`), `loadDocument()`/`loadCollectionWithQuery()`, `onSubmit`,
  persistence, `autoSave: true`. **NEVER `sendResponse()`d for a read/display screen** (Home, My
  Reports, detail) — those go through the Display Doc's custom cards.
- **Display Doc** (`reportDisplayDoc`) — owns ONLY CardsSet sections (Home, My Reports list, Detail
  cards, timeline). No Field objects, no Collections. `sendResponse()` renders this Doc. HTML reads
  field values from the Data Doc.

> **The editable submit form is the one exception (resolves the rule-8 / "render inline" ambiguity).**
> The submit form is a set of standard `FormFieldTypes` sections on the Data Doc, and the framework's
> only way to render an *editable* framework form is `reportDoc.sendResponse()`. So the submit-form
> navigation intent renders the Data Doc directly — this is correct, not a rule-8 violation. Rule 8
> governs the *read/display* screens (custom HTML cards), which must route through the Display Doc.
> This mirrors the shipped sailors-cart pattern: the Add/Edit form renders via `productDoc.sendResponse()`
> while the per-screen Display Doc stays an empty `tabBarHidden` shell (see
> `sailors-admin/src/docs/product-edit-display.js`). The Display Doc shell for the submit form is
> optional — anonymous-user renders the submit form straight off `reportDoc`.

**Why Two-Doc:** `forCollection` sections CANNOT carry a CardsSet (framework constraint). Two Docs
give one consistent separation for all sections. Standard form sections (Submit) still live on the
Data Doc and the framework renders them inline.

## Per-platform rendering (REQUIREMENTS §9.1) — project-specific

Every `custom_card` / `display_only` Display section is a folder `src/sections/display/<screen>/`:
- `index.js` — the `Section` + `CardsSet`; its `onResponse` reads prepared data from a
  `state.setField` stash and calls `renderForPlatform(data, { web: renderWeb, mobile: renderMobile })`
  (one dispatch, no other branching).
- `web.js` / `mobile.js` — full per-platform markup. Shared **theme tokens** in `lib/utils/theme.js`;
  shared `escapeHtml` + builders in `lib/utils/format.js`. Detect platform once via `state.client`
  against `ALL_CONSTANTS.CLIENTS.*` wrapped in `lib/utils/platform.js`.

## rendering: standard
- `Section` with `fields:` on the Data Doc; framework renders inline via `FormFieldTypes`.
- No CardsSet, no custom HTML.
- **Doc ref:** `docs/frontm-ai-doc-field-section-collection-data-modeling-guide.md`.

## rendering: custom_card (regular section)
- **Data Doc section:** `Section` + Field objects (persistence). Never rendered.
- **Display Doc section:** `Section` + `CardsSet` + `CARD_TYPES.HTML`. No Field objects.
- `Card.content` is an HTML string built in the display section's `onResponse` (via
  `renderForPlatform`) from Data Doc field values; `Card` is `readOnly: true`.
- Edit/action button: `<button data-action="intent" data-intent-id="[intent]" data-payload='{...}'>`.
- The intent's `onResolution` opens the Data Doc section as a popup via `sendQuickFormResponse()`;
  on save, `onSubmit` calls `self.collection.parentDoc.save()` then re-renders the Display Doc.
- **Doc refs:** `docs/frontm-ai-cards-cardsets-comprehensive-guide.md`,
  `docs/frontm-ai-intent-class-events-lifecycle-reference.md`.

## rendering: custom_card (sub-collection) — e.g. `amendments`
- **Data Doc section:** `Section({ forCollection: true })` + `addCollection(...)`. Owns the rows.
  NO CardsSet.
- **Display Doc section:** `Section` + `CardsSet` + `CARD_TYPES.HTML`. Renders an HTML table of rows.
- Add button: `data-action="intent"` → `add_intent`. (Amendments are **append-only** — no edit/delete
  intents; original rows are immutable.)
- `add_intent` loads a focused sub-entity Doc and calls `sendQuickFormResponse()` (Doc needs
  `confirm`/`cancel`; fields need `includeInQuickEdit: true`).
- **Doc refs:** `docs/frontm-ai-cards-cardsets-comprehensive-guide.md`,
  `docs/frontm-ai-collection-class-comprehensive-guide.md`.

## rendering: display_only — e.g. `statusHistory` timeline, Home, Resolution
- **Data Doc** owns any persisted fields (or, for `statusHistory`, a `forCollection` sub-collection
  written by the transition path). **Display Doc section:** `Section` + `CardsSet` + `CARD_TYPES.HTML`.
- `Card.content` built in `onResponse` only; no edit buttons, no intents.
- **Doc ref:** `docs/frontm-ai-cards-cardsets-comprehensive-guide.md`.

## rendering: standard (sub-collection)
- Not used in anonymous-user. Where used: `Section({ forCollection: true })` + `addCollection(...)`;
  framework table + built-in `confirmAction` popup. No CardsSet.

## Code structure — file separation
- `src/docs/report-doc.js` — Data Doc (constructor only).
- `src/sections/<section>.js` — Data Doc Section + Field objects.
- `src/sections/display/<screen>/{index,web,mobile}.js` — Display Doc Section + CardsSet + per-platform renderers.
- `src/frames/<intent>.js` — intent handlers (popups, transitions, navigation).
- `src/frames/contracts/<msg>.js` — `onMatching` handlers for inbound bot-to-bot messages.
- `src/main.js` — main intent, `state.onConfig`, side-effect imports only.

## Framework Construct Checklist

### Doc properties by usage context
| Usage context | Required | Why |
|---|---|---|
| Doc used with `sendQuickFormResponse()` (Amend popup) | `confirm: "Save"`, `cancel: "Cancel"` | Otherwise the popup has no buttons. |
| `reportDoc` (draft autosave) | `autoSave: true` | U-F9. **Triggers the Context bootstrap** (rule 17/22). |
| Display Doc, single-tab app | no `title`; `tabBarHidden = true` before `sendResponse()` | Avoid triple-title; hide empty tab bar. |

### Field properties by data semantics
| Semantics | Required | Why |
|---|---|---|
| Evidence upload (`evidenceFile*`, `amendmentEvidenceKey`) | `type: FILE_FIELD` | Upload control + S3. Never TEXT/URL. |
| `contactValue` (reporter-private) | `encrypted: true` | Client-level encryption (sensitive). |
| Field in a popup (`amendmentNote`, `amendmentEvidenceKey`) | `includeInQuickEdit: true` | Else excluded from `sendQuickFormResponse()`. |
| `reportId` | `primaryKey: true`, `hidden: true` | Unique per row; document lookup. |
| `reporterId` | `hidden: true`, NOT `primaryKey` | Per-user filter + ownership; many rows share it. |

### Section properties by rendering
| Pattern | Required | Why |
|---|---|---|
| `forCollection: true` (`amendments`, `statusHistory`) | NO CardsSet on this section | Framework constraint — CardsSet lives on the Display Doc. |
| Display Doc card section | `borderless: true`, `grid: { row, column }` | Custom HTML renders its own chrome; positions the card. |

### HTML Card properties
| Context | Required | Why |
|---|---|---|
| Card with action buttons | `readOnly: true` on the Card | Else the whole surface intercepts clicks. |
| Card showing an evidence FILE_FIELD value | signed URL via `state.frontmlib.getS3SignedUrl(bucket, "${state.conversationId}/${key}", SIGNED_URL_EXPIRY_SECONDS)` built **before** `sendResponse()` (NOT in `onResponse`); bucket via `state.getStaticData(STATIC_DATA_KEYS.CONVERSATIONS_BUCKET)` | `.value` is an S3 key, not a URL; render handlers aren't awaited. See `docs/frontm-ai-s3-signed-urls-frontmlib-guide.md`. |

### Data loading requirements
| Context | Required in app-start | Why |
|---|---|---|
| My Reports list | `reportsCollection.loadCollectionWithQuery({ query: { reporterId } })` before `sendResponse()` | Scope to the current reporter; without `reportId` PK the collection dedupes rows. |
| Report detail | `reportDoc.loadDocument({ reportId })` + ownership assertion | Else empty Doc / cross-user leak. |
| Sub-collections (`amendments`, `statusHistory`) | `collection.loadCollectionWithQuery({})` after `loadDocument()`, before `sendResponse()` | `loadDocument` does not auto-populate `collection.rows`. |

## Code generator rules

1. **Composing documented primitives is NOT inventing.** Build each `rendering` value from the
   primitives above; only ask when a primitive itself is missing from `docs/`.
2. **Never silently downgrade.** If the spec says `custom_card`, never emit `standard` instead — STOP
   and ask.
3. **Pre-flight check.** For every section, confirm the generated code honours its `rendering`.
4. **Two-Doc.** Data Doc owns fields/collections/persistence (never `sendResponse()`d); Display Doc
   owns CardsSet (the only Doc `sendResponse()`d).
5. **Fields always required for persistence.** Data Doc declares a Field for every persisted value;
   Display Doc has none.
6. **One file per concern** (see "Code structure" above).
7. **`forCollection` sections CANNOT have CardsSet.** Data Doc owns `forCollection`; Display Doc has
   the separate CardsSet section.
8. **`sendResponse()` on the Display Doc for every read/display screen** (Home, My Reports, detail).
   App-start loads the Data Doc, then sends the Display Doc. The read-screen **navigation intents**
   (`openMyReports`, `openReportDetail`) MUST also render via `reportDisplayDoc.sendResponse()` after
   loading — never the Data Doc. **Sole exception:** the editable *submit* form renders via
   `reportDoc.sendResponse()` (standard `FormFieldTypes` sections; see the Two-Doc note above) — there
   is no other way to show an editable framework form. Per-screen visibility on the single shared
   `reportDisplayDoc` is by `section.hidden` (documented:
   `docs/frontm-ai-doc-field-section-collection-data-modeling-guide.md` § "Common Section Properties").
9. **Per-platform renderers (§9.1).** Every Display section dispatches once via
   `renderForPlatform(data, { web, mobile })`; no scattered `isMobile()` ternaries; theme tokens in
   `lib/utils/theme.js`, `escapeHtml`/builders in `lib/utils/format.js`.
10. **Sanitise all free-text before HTML/email** (`description`, `accusedParty`, `evidenceNotes`,
    `resolution`, `rejectReason`, `amendmentNote`, `note`) via `lib/utils/format.js` `escapeHtml` /
    `lib/validation.js` sanitiser — NFR-2.
11. **Evidence signed URLs** — build server-side BEFORE `sendResponse()`, never in `onResponse`;
    bucket from static data, never hardcoded (A-F7, S3 rule).
12. **Optimistic concurrency on every transition** — re-read by `reportId`, validate the move against
    current `status`+`version` via `lib/ticket-status.js`, reject stale writes (ER-B5). Append a
    `statusHistory` row in the same path.
13. **Idempotent jobs/claims** — status + job-id/`attendedBy`/`version` conditional so duplicate/stale
    fires are safe no-ops (ER-B8). (Admin app.)
14. **Routing only through `resolveAssignees(report)`** (`lib/access.js`, D17) — no hardcoded role
    queries anywhere.
15. **Single admin-read gateway** — `loadReportsForAdmin()`/`loadReportForAdmin()` always apply
    `adminProjection`; no admin code path queries `reports` directly (ER-A3). (Admin app.)
16. **Identity-free payloads.** No bot-to-bot payload or admin email/HTML ever carries `reporterId`,
    `contactMethod`, `contactValue`, or an `actorId` (use `actorRole`). Send AFTER `save()`.
17. **autoSave bootstrap (CRITICAL).** Because `reportDoc` is `autoSave: true`, `src/main.js` MUST set
    `state.onConfig = () => { state.contextAware = true; }` AND `main.onResolution` MUST call
    `await Context.CreateAndInit("mainApp", { state })` BEFORE any `loadDocument`. Missing this
    silently disables every autoSaveBuffer write (CLAUDE.md "App Entry-Point Bootstrap").
18. **Media-field envelope.** `evidenceFile*`/`amendmentEvidenceKey` `.value` is
    `{ value: <s3-key>, fileName, fileScopeValue }` — drill into `.value?.value` for the key; never
    embed the raw key in HTML.
19. **Constants only.** Every key/enum/error-code/static-data-key is a `SCREAMING_SNAKE_CASE` constant
    (`lib/constants.js` or `src/constants.js`) — no magic strings/numbers.
20. **Two execution contexts.** Inbound `onMatching` contract handlers and `delete`/transition intents
    run in **Context B** (empty graph) — `loadDocument`/`loadCollectionWithQuery` before reading.
    Pass B→B state via `setField`/`setPersistedField` (sandbox-safe for inbound messages).
21. **Context-B sub-entity access.** A Context-B handler that needs a sub-collection MUST first
    `await reportDoc.loadDocument({ reportId })`, then reach the sub-collection via the **loaded
    parent** (`reportDoc` after load IS hydrated) — NOT via the imported sub-collection module
    singleton. (CLAUDE.md "Collection and parent-Doc access" → Context-B variant.)
22. **Context.Create vs CreateAndInit; no clone for add.** `Context.CreateAndInit("mainApp",{state})`
    is called ONCE in `main.onResolution` (wipes buffer, runs `onInit`). Every subsequent popup
    intent (Amend) attaches via `Context.Create(state.currentTabId,{state})` — preserves the buffer,
    NO re-`loadDocument`. For Add, set a new sub-entity `docId` (`state.getUniqueId()`) FIRST, then
    clear field values in place — **never** `cloneAndInit()` (the clone shares `intentId`, is silently
    un-registered, and the submit lands on the original → empty row, no persistence). See rule 26.
23. **Shell UI flags in `state.onConfig`, mirrored from BRD §8.1.** Every BRD §8.1 row whose Setting ≠
    "platform default" gets one `state.onConfig` line; every `state.onConfig` assignment has a BRD §8.1
    row. anonymous-user: `state.contextAware = true` (only non-default row). Never set shell flags in
    `main.onResolution`/handlers, never via CSS, never invent `domain.json` flags.
24. **Conditional fields are code, not declarative.** `contactValue` show/require-by-`contactMethod`
    (U-F7) is enforced in `onValidation`/`onSave`, not a non-existent "show-if" property.
25. **Append-only sub-collections** (`amendments`, `statusHistory`) emit an `add` path only — no edit
    or delete intent; existing rows are immutable (D16; history integrity).
26. **Add-intent reset pattern (clone failure mode).** For `addAmendment`: `await Context.Create(...)`,
    set `amendmentSubDoc.docId = state.getUniqueId()`, then loop `field.value = null`, then
    `sendQuickFormResponse()`. The `cloneAndInit()` path is WRONG — the clone inherits the original's
    `intentId`, `state.setIntentWithId` silently un-registers it, the submit dispatch lands on the
    original Doc, and the new row neither appears in the table nor persists to MongoDB.

### anonymous-admin-specific rules

27. **Access gate runs in Context B BEFORE the bootstrap (A-F1).** `main.onResolution` first verifies
    `isAdmin()` / `resolveAdminRole()` against the seeded `admin-users` registry. On deny: render the
    refusal card and STOP — no `Context.CreateAndInit`, no gateway read. On allow: `state.setField` the
    role (for A-F4), then `await Context.CreateAndInit("mainApp", { state })`, then render the
    Dashboard. (`adminReportDoc` is `autoSave: true`, D-L3-2 → the bootstrap is mandatory, rule 17.)
28. **Small-cell suppression (A-F2, ER-A6, D-L3-3).** Every dashboard breakdown that could
    de-anonymise a tiny population (per-ship especially) MUST merge any cell with **fewer than 5**
    reports into "Other"/suppressed. The threshold `5` is a `SCREAMING_SNAKE_CASE` constant
    (`lib/constants.js`, rule 19), not a magic number. Aggregations run over `loadReportsForAdmin`
    (projection applied); **no Atlas Charts** (D4) — custom HTML stat cards only.
29. **Per-action transition popups (A-F8–F12, D-L3-4).** `resolveReport`/`escalateReport`/
    `closeRejected`/`overrideSeverity` each open a focused `sendQuickFormResponse()` popup on its **own
    per-action CAPTURE Doc** — `resolveCaptureDoc` / `severityCaptureDoc` / `noteCaptureDoc`
    (`autoSave: false`; `confirm`/`cancel` set on the constructor; the single capture field
    `includeInQuickEdit: true`) — **NOT** on the shared `adminReportDoc`. `takeReview` is a direct
    transition (no popup).

    **Why capture Docs, not the shared Doc (MP-FIX-ADMIN-POPUP-CAPTURE-DOCS, 2026).** A `Doc` has
    exactly ONE `onSubmit` slot (`Doc.js` — fired on the `CONFIRM` action), and `sendQuickFormResponse()`
    selects fields only by `includeInQuickEdit` (+ optional `currentScope`/`field.scopes`). Binding every
    popup to `adminReportDoc` therefore breaks two ways: (a) each EDIT frame assigning
    `adminReportDoc.onSubmit = …` as a side-effect import means the **last import silently clobbers its
    siblings** (5 handlers — resolve/escalate/closeRejected/overrideSeverity + the A-F13 manual-log submit
    — would contend for one slot); (b) the three popup fields all carry `includeInQuickEdit: true` with no
    discriminator, so **any one popup renders `resolution` + `severity` + `transitionNote` together**.
    Separate capture Docs give each action its own `onSubmit` and an isolated one-field form — the same
    proven pattern as the user app (`rejectReasonDoc` / `amendmentDoc`), and consistent with rule 30's
    "one `(doc, section)` home" discipline.

    **Flow.** The capture Doc holds the operator's input only. Its `onSubmit` (Context A on confirm)
    re-reads the report FRESH (`Context.Create` attach + `adminReportDoc.loadDocument({ reportId })` —
    rule 22; the trigger intent stashed `reportId` via `state.setField`), re-resolves the caller's admin
    ROLE authoritatively (`resolveAdminRole`), applies the concurrency + legality guard
    (`canTransition(current, …, role)` against the just-read status — rule 12), copies the **sanitised**
    capture value onto the report's own **hidden infra column** (`resolution` / `severity` are hidden,
    read-only-on-form infra fields on `adminReportDoc`, alongside `rejectReason`), appends ONE
    `statusHistory` row (rule 12), `adminReportDoc.save()`s, then fires any `MSG_*` **after** `save()`
    (rule 16). The escalate/close note is a **transient `transitionNote` capture field** — consumed into
    `statusHistory.note`, **never persisted as a `reports` column** (`noteCaptureDoc` field has no
    `dbName`).
30. **Admin Doc binds NO reporter identity (C1, ER-A2/A3).** `adminReportDoc` declares no `reporterId`/
    `contactMethod`/`contactValue` field and no reporter-create audit field. The manual-log form
    (A-F13) has **no contact fields** (D-L3-5; `source=MANUAL`, `reporterId` empty). Amendments are
    **read-only** on the admin side (no add/edit/delete intent — the reporter owns U-F13). The
    `admin-users` availability control is a 3-state value field (`available`/`busy`/`unavailable`) —
    a DROPDOWN driven by buttons, **never a SWITCH** (which is boolean); only the caller's own
    `availability` row is writable. Call answering (A-F21) is an **atomic claim** — first writer of
    `ACTIVE`/`attendedBy` wins (rule 13), then `MSG_CALL_STOP_RING` to the others.

## Anti-patterns
- ❌ Fields + CardsSet on the same Doc. ✅ Fields on Data Doc, CardsSet on Display Doc.
- ❌ CardsSet on a `forCollection` section. ✅ separate Display Doc section.
- ❌ `sendResponse()` on the Data Doc. ✅ Display Doc only.
- ❌ `isMobile() ? … : …` inside a renderer. ✅ one `renderForPlatform` dispatch + `web.js`/`mobile.js`.
- ❌ Raw S3 key in `<img>/<a>`. ✅ signed URL built before `sendResponse()`.
- ❌ `cloneAndInit()` to add a sub-entity row. ✅ new `docId` + clear values in place (rules 22/26).
- ❌ Re-`loadDocument` in a popup intent. ✅ `Context.Create` (preserve buffer) (rule 22).
- ❌ Hardcoded role query for routing. ✅ `resolveAssignees` (rule 14).
- ❌ Any reporter-identity field bound on an admin Doc / in a payload. ✅ `adminProjection` + `actorRole` (rules 15/16).
