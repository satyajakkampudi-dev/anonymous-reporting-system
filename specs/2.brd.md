## 1. Purpose

People who witness misconduct — harassment, safety violations, fraud/ethics breaches,
bullying/retaliation — need a channel they trust enough to actually use. Existing channels
(phone, email, word-of-mouth) expose the reporter, lose reports, and give no visibility into the
outcome. This system gives internal crew and shore staff a **safe, code-enforced-anonymous**
channel to submit and **track** reports to resolution, while a central compliance team triages,
resolves, and escalates them **without ever seeing who reported**. Anonymity is enforced in code
(a single `adminProjection` read gateway, identity-free cross-app payloads, masked voice calls),
not assumed from the platform.

## 2. Scope

**In scope (v1).** Two frontm.ai micro-apps over one shared MongoDB cluster and one shared `/lib`:

- **anonymous-user** — submit a report + evidence, track own reports, accept/reject a resolution,
  withdraw, amend, and place an anonymous voice call (with voicemail fallback).
- **anonymous-admin** — role-gated triage queue (identity-stripped), dashboard + custom-HTML
  analytics, full status lifecycle, priority surfacing of CRITICAL/escalated reports, manual
  logging, auto-escalation / auto-close / SLA-digest jobs, on-call availability + call answering,
  and CSV/PDF case export.

Both apps run on **mobile and web** with separate per-platform renderers dispatched once via
`state.client` (REQUIREMENTS §9.1).

**Explicitly excluded from v1** — see §10.

## 3. Actors

| Actor | Role | Location | Platform | Organisation |
|---|---|---|---|---|
| **Reporter** | Authenticated internal crew (any vessel) or shore staff; submits + tracks own reports | At sea / shore | mobile + web | Same org |
| **Primary admin** (`PRIMARY_ADMIN`) | Central shore compliance officer; triage → resolve / escalate; manual log; analytics | Shore | mobile + web | Same org (central team) |
| **Secondary admin** (`SECONDARY_ADMIN`) | Senior / escalation (e.g. Designated Person Ashore); everything Primary does **plus** escalated + against-admin reports | Shore | mobile + web | Same org (central team) |
| **System** (job scheduler) | Non-human: auto-escalate, auto-close, SLA-backstop digest timers; fires intents inside anonymous-admin | — | `state.jobScheduler` | — |

**Anonymity boundary (D17).** Admins are a *single central shore team* (`scope = GLOBAL`),
deliberately **not** on the reporter's vessel. `shipName`/`location` are incident metadata, never
routing keys. Per-vessel admins are explicitly rejected.

## 6. Cross-App Contracts

> Defined once in `lib/constants.js`. **No payload carries reporter identity.** This table is the
> primary coordination artefact between the two app teams; it is reproduced standalone in §11.

| Contract | Sender | Receiver | Message type | Payload (identity-free) | Receiver action |
|---|---|---|---|---|---|
| New report | anonymous-user (U-X1) | anonymous-admin (A-X1) | `MSG_NEW_REPORT` | reportId, category, urgency, severity, assignedTo, createdOn | Notify assigned admins; schedule auto-escalate (D2) |
| Reopened | anonymous-user (U-X2) | anonymous-admin (A-X2) | `MSG_REPORT_REOPENED` | reportId, reopenCount, rejectReason | Notify assigned admins |
| Resolved | anonymous-admin (A-X4) | anonymous-user (U-X4) | `MSG_REPORT_RESOLVED` | reportId, resolvedOn | Notify reporter (email + push) |
| Status changed | anonymous-admin (A-X5) | anonymous-user (U-X5) | `MSG_REPORT_STATUS_CHANGED` | reportId, newStatus | Notify reporter |
| Closed | anonymous-admin (A-X6) | anonymous-user (U-X6) | `MSG_REPORT_CLOSED` | reportId, closeType | Notify reporter |
| Incoming call | anonymous-user (U-X3) | anonymous-admin (A-X3) | `MSG_INCOMING_CALL` | callRef, meetingId (+ VoIP push) | Ring available admins |
| Ring-stop | anonymous-admin (A-X7) | anonymous-admin (A-X7) | `MSG_CALL_STOP_RING` | callRef, meetingId | Stop ringing the other admins |

**Incoming-message routing.** Each receiving app needs an `onMatching` handler per inbound type:
anonymous-admin handles `MSG_NEW_REPORT`, `MSG_REPORT_REOPENED`, `MSG_INCOMING_CALL`,
`MSG_CALL_STOP_RING`; anonymous-user handles `MSG_REPORT_RESOLVED`, `MSG_REPORT_STATUS_CHANGED`,
`MSG_REPORT_CLOSED`. These run in **Context B** (empty object graph) — load any report by `reportId`
before reading it.

## 7. Custom Frame Descriptions

> Standard frames (U-F0–F3, A-F0/F3/F6 CRUD scaffolding) follow the inferred frontm.ai pattern and
> are not detailed here. Below are the frames carrying PM-confirmed business rules.

### anonymous-user

#### U-F0a: Access gate (reporter licence)
- **Trigger:** app start.
- **Business rule (FR-U0):** verify the caller holds the FrontM platform licence/role
  `quitelineenduser` (`state.user.roles`). **Pure licence-role check — no registry fallback**
  (unlike the admin gate A-F1, the reporter app has no curated allow-list collection).
  `Context.CreateAndInit` runs FIRST (it creates the tab / `state.currentTabId` that any render
  needs — see framework-mapping rule 31), THEN the gate. On miss: render the refusal **through the
  Display Doc** (`showScreen(REFUSAL)` → `reportDisplayDoc.sendResponse()`) and STOP **before** the
  reports load. On allow: continue (load reporter's reports → render Home). Refusal copy is static
  (no field values, no network) — safe on a cold start and a poor maritime link.
  ⚠️ The refusal MUST render via the Display Doc, NOT a standalone CardsSet sent pre-Context — that
  shows a blank screen (MP-FIX-ACCESS-REFUSAL-RENDER).
- **Affected fields:** reads `state.user.roles` only; no write.

#### U-F5: Pre-submit anonymity guard
- **Trigger:** reporter is about to submit (and on opening the form).
- **Business rule (ER-A1):** show an anonymity warning + a "what the admin will see" preview built
  from the `adminProjection` field set, plus guidance not to include self-identifying detail
  (small crews can be de-anonymised by free-text + ship + date + accused even with keys stripped).
- **Affected fields:** read-only render of all admin-visible fields; no write.

#### U-F6: Evidence upload validation + atomicity
- **Trigger:** file attached / form save.
- **Business rule (D1, FR-U2, ER-C10):** ≤ 5 files, ≤ 25 MB each; allow images, PDF, doc/docx,
  audio, video, text; validate extension **and** content type server-side; reject otherwise;
  no orphaned S3 objects and no saved-without-evidence partial state.
- **Affected fields:** `evidenceFile1..N`, `evidenceNotes`.

#### U-F7: Contact-method conditional validation
- **Trigger:** contact method selected; form save.
- **Business rule (FR-U1, SPEC):** `None` hides the value; `Email` → RFC-pragmatic regex;
  `Phone` → E.164-tolerant; `Cabin number` → alphanumeric pattern. Contact fields are
  reporter-private (excluded from `adminProjection`).
- **Affected fields:** `contactMethod`, `contactValue`.

#### U-F8: Submit-time transforms + idempotency
- **Trigger:** submission Doc `onSubmit`.
- **Business rule (D17, ER-B9, SPEC):** sanitise all free-text; `incidentDate` must parse and not
  be in the future; generate `reportId` (`RPT-` + 10 chars) and **retry on unique-index
  violation**; set `status = OPEN`, `createdOn`, `source = REPORTER`; route via
  `resolveAssignees(report)` (againstAdmin → SECONDARY_ADMIN, else PRIMARY_ADMIN; scope GLOBAL);
  debounce double-submit.
- **Affected fields:** `reportId`, `status`, `assignedTo`, `createdOn`, `source`, all free-text.

#### U-F9: Draft autosave
- **Trigger:** field edits on the submission form (Doc `autoSave: true`).
- **Business rule (D14, ER-C13):** the in-progress submission survives crash/navigation on slow
  maritime links; limited/resumable uploads. **Requires** the Context bootstrap
  (`state.contextAware = true` + `Context.CreateAndInit` in `main.onResolution`) per CLAUDE.md.
- **Affected fields:** all submission fields (buffered to Redis autoSaveBuffer).

#### U-F10 / U-F11 / U-F12: Reporter transitions (accept / reject / withdraw)
- **Trigger:** action button on a report detail.
- **Business rule (FR-U6, D10, D16, ER-B5/B6):** Accept → `CLOSED_BY_USER`. Reject (reason) →
  `REOPENED`, `reopenCount` 0→1 **once only**; after the cap the reject action is unavailable.
  Withdraw (only `OPEN`/`UNDER_REVIEW`) → `WITHDRAWN`. Each re-reads + validates the transition
  against current `status`+`version` (optimistic concurrency); ownership asserted first.
- **Affected fields:** `status`, `reopenCount`, `rejectReason`, `withdrawnOn`, `version`,
  `updatedOn`.

#### U-F13: Amend / append (append-only)
- **Trigger:** "Amend" on a non-terminal report.
- **Business rule (D16, ER-C11 — PM choice: append-only popup):** a quick-form appends a
  timestamped, audited note (and optionally new evidence) to the report; the original
  reporter-entered fields stay locked. Avoids duplicate reports without rewriting history.
- **Affected fields:** an append-only amendment note collection/notes field + `evidenceFile*`;
  audit fields.

#### U-F14: Reporter notification dispatch
- **Channel (rule 32):** mobile + web push (`sendPushToCurrentUser` → `sendPushNotifications` on the
  reporter's own conversation) + email. notifyReporter runs in the reporter's own session (submit
  onSubmit; X4/X5/X6 receivers), so push-to-self reaches their device and browser. Identity-free
  push `data` (`{ reportId }`).
- **Email template (rule 33):** body built via the shared `lib/email-template.js` (`renderEmail`) —
  text wordmark, table layout, inline styles, NO tracking pixel / NO remote images, identity-free.
- **Trigger:** own action (submit/accept/reject/withdraw) and inbound `MSG_REPORT_*`.
- **Business rule (FR-U8, notifications matrix §11):** email + web push on received, status
  changed, resolved, closed. Best-effort but logged (NFR-4).
- **Affected fields:** none (notification side-effect).

#### U-F15 / U-F16 / U-F17: Anonymous calling (user side)
- **Trigger:** "Call compliance (anonymous)"; 30 s ring timeout; hang-up/drop.
- **Business rule (FR-C1/C3/C5/C7, D7/D8/D12, ER-A5/C12):** create a voice-only Daily.co meeting
  with a **masked/system host** (never `state.user.userEmail`); reporter joins as a per-call
  throwaway masked guest ("Anonymous Reporter"), video muted; create an identity-free `RINGING`
  `call-queue` entry; ring `available` admins via `MSG_INCOMING_CALL` + VoIP (callRef + meetingId
  only). On 30 s no-answer → `MISSED` → voicemail (≤ 3 min / 25 MB) → auto-create report
  (`source = CALL`, voicemail as evidence, `reporterId` empty) → `MSG_NEW_REPORT`. Reporter hangs
  up before answer → `ABANDONED`; mid-call drop → inactivity timeout `ENDED`. **No recording.**
- **Affected fields (call-queue):** `callRef`, `status`, `meetingId`, timestamps, `voicemailKey`,
  `linkedReportId`.

### anonymous-admin

#### A-F1: Access gate
- **Trigger:** app start (Context B, before any read).
- **Business rule (FR-A1, D3):** verify platform licence/role **and** presence in the seeded
  `admin-users` registry; refuse non-admins with a clear message. The registry is the single
  source for gating, role, recusal, and availability.
- **Affected fields:** reads `admin-users`.

#### A-F2: Dashboard aggregation
- **Trigger:** dashboard render (entry).
- **Business rule (FR-A2/A9, D4, ER-A6):** counts by status / severity / age from live
  aggregation via `loadReportsForAdmin` (projection applied); rendered as **custom HTML** stat
  cards (no Atlas Charts); **small-cell suppression** on any breakdown that could de-anonymise a
  tiny population (e.g. per-ship counts below threshold). Includes a Priority/Escalated card
  feeding A-F5.
- **Affected fields:** aggregate reads only.

#### A-F4: Role-filter + recusal
- **Trigger:** queue assembly.
- **Business rule (FR-A3, D9, ER-A4):** Primary → OPEN/UNDER_REVIEW; Secondary → ESCALATED +
  against-admin. A report flagged *about* a specific admin is hidden from that admin and routed
  away. ⚠️ **Accepted residual gap (D9):** a report about the **secondary** admin is still visible
  to them in v1 — revisit when a designated alternate/ombudsperson exists.
- **Affected fields:** queue filter over `status`, `assignedTo`, `accusedParty`/`againstAdmin`.

#### A-F5: Priority surfacing & filtering  *(added at PM request)*
- **Trigger:** queue + dashboard render.
- **Business rule (FR-A2/A3/A9, D6, ER-B7):** reports with **severity = CRITICAL**, **urgency =
  Immediate risk**, or **status = ESCALATED** are (a) visually differentiated — distinct
  status/severity tone tokens from `lib/utils/theme.js`, a priority badge, and sorted to the top of
  the queue — and (b) available as a **"Priority / Escalated" quick-filter** on the queue with a
  matching stat card on the dashboard. Severity reflects admin override (A-F12). Reinforces that
  escalation never rots (ER-B7) by keeping breaching items at eye level. **Confirmed (PM): both
  visual highlight + quick-filter.**
- **Affected fields:** read-only over `severity`, `urgency`, `status`; render only.

#### A-F7: Evidence signed URLs
- **Trigger:** report detail render.
- **Business rule (FR-A4, NFR-2, S3-signed-URL rule):** evidence S3 keys are signed **server-side
  before `sendResponse`** (not in `onResponse`, which is not awaited); short-lived; re-fetch on
  expiry (ER-D16). "Download at your own risk" note (D13). Bucket read from static data, never
  hardcoded.
- **Affected fields:** `evidenceFile1..N` (envelope `.value` is the S3 key).

#### A-F8 / A-F9 / A-F10 / A-F11: Admin transitions
- **Trigger:** action buttons on report detail.
- **Business rule (FR-A5, ER-B5):** take review → UNDER_REVIEW; resolve → RESOLVED (+
  `MSG_REPORT_RESOLVED`, schedule auto-close +30 d); escalate → ESCALATED + `assignedTo =
  SECONDARY_ADMIN` + notify secondary; close-rejected (REOPENED → CLOSED_REJECTED). Every
  transition re-reads the report and validates against current `status`+`version`; stale writes
  rejected and surfaced, not overwritten. All audited (admin identity only). Sends
  `MSG_REPORT_STATUS_CHANGED` on status changes the reporter should see.
- **Affected fields:** `status`, `assignedTo`, `resolution`, `resolvedOn`, `version`, `updatedOn`,
  audit fields.

#### A-F12: Severity override
- **Trigger:** triage.
- **Business rule (D6):** initial severity is derived from reporter urgency; the admin may override
  it. Override feeds priority surfacing (A-F5) and auto-escalate timing (CRITICAL +1 d).
- **Affected fields:** `severity`.

#### A-F13: Manual log
- **Trigger:** "Log a manual report".
- **Business rule (FR-A6, §7.7):** Form Doc → `source = MANUAL`, `reporterId` empty (no
  notifications/tracking owner), `OPEN`, routed via `resolveAssignees`. Omits reporter-driven
  transitions and reporter notifications (SPEC concurrency note).
- **Affected fields:** the report content fields; `source`.

#### A-F14: Case export
- **Trigger:** "Export" on a report or filtered set.
- **Business rule (D15):** CSV / PDF export of report content **from the `adminProjection` set
  only** (no reporter identity in the export). PDFs via the HTML class / `toPDF`.
- **Affected fields:** read-only over the admin-visible set.

#### A-F15: Admin notification dispatch
- **Trigger:** inbound `MSG_NEW_REPORT` / `MSG_REPORT_REOPENED`; escalation; auto-escalate job; manual-log.
- **Business rule (FR-A10, matrix §11):** email + **mobile/web** push to assigned admins on
  new/reopened (assigned) and to secondary admins on escalation. Best-effort but logged; the SLA
  digest (A-F18/F19) is the fallback so a failed notification never means an unseen report (ER-D15).
- **Channel + delivery (rule 32):** mobile+web push requires push-to-self in the recipient's own
  session. The `MSG_NEW_REPORT` (X1) and `MSG_REPORT_REOPENED` (X2) receivers already run there →
  they push-to-self (`sendPushToCurrentUser`) + email-to-self. Escalate / auto-escalate / manual-log
  run in the acting admin's session or a job (recipients remote), so they DISPATCH `MSG_ADMIN_NOTIFY`
  (`{ reportId, event }`) to the resolved assignee `userId`s; the `adminNotifyReceiver` runs in each
  recipient's session and push-to-self + email-to-self. A job has no user session, so this dispatch
  is the only way it can reach admins on mobile.
- **Email template (rule 33):** body built via the shared `lib/email-template.js` (`renderEmail`),
  same shell as the reporter email — text wordmark, table layout, inline styles, NO tracking pixel /
  NO remote images; the detail rows stay identity-free.
- **Affected fields:** none (side-effect).

#### A-F16 / A-F17 / A-F18: System jobs
- **Trigger:** `state.jobScheduler` fires (Context B intents).
- **Business rule (FR-A7/A8, D2, D11, ER-B7/B8):**
  - **Auto-escalate** (scheduled at submit: CRITICAL +1 d, others +3 d) — if still OPEN/unactioned
    → ESCALATED + notify secondary; else idempotent no-op + log. The transition lives in the shared
    `escalateOpenReport(reportId)` so the SLA sweep can perform it too (see below + framework-mapping
    rule 34).
  - **Auto-close** (scheduled at resolve: +30 d) — if still RESOLVED → CLOSED_BY_SYSTEM +
    `MSG_REPORT_CLOSED`; else no-op + log.
  - **SLA backstop** (OPEN 24 h / ESCALATED 24 h) — email digest of breaching reports to **all**
    admins, AND **performs escalation** for any OPEN report past its escalate deadline that the
    scheduled job never armed (the timer is armed only via the X1 receiver, which is skipped when no
    admin is seeded — so the sweep is the assignee-independent backstop; MP-FIX-ESCALATE-DECOUPLE,
    rule 34). Guarded → a report the job already escalated is a no-op here.
  - All guarded by current `status` + a job-id/`version` conditional so stale/duplicate fires are
    safe no-ops (no double-escalate/close/notify).
- **Affected fields:** `status`, `version`, `updatedOn`.

#### A-F19: In-app alerts / digest screen
- **Trigger:** admin opens "Alerts".
- **Business rule (ER-D15, D11):** a Display listing SLA-breaching reports + any
  notification-fallback items, so a missed email still surfaces in-app. Identity-stripped.
- **Notification-failure bridge (REQUIRED — see MP-FIX-ALERTS-FAILURE-BRIDGE):** A-F15 records
  delivery failures to a DURABLE cross-admin shared field (`SHARED_KEYS.NOTIFICATION_FAILURES`);
  the alerts renderer reads a SYNCHRONOUS per-render state stash (`STATE_KEYS.NOTIFICATION_FAILURES`)
  because `section.onResponse` is not awaited. These are two different fields — the screen only
  shows failures if app-start (Context B, before render) reads the shared field, prunes expired
  entries, and writes the render stash. Without that bridge the banner is permanently empty even
  when failures exist (the safety net is inert).
- **Affected fields:** read-only over the admin-visible set.

#### A-F20 / A-F21 / A-F22: Calling (admin side)
- **Trigger:** availability toggle; inbound `MSG_INCOMING_CALL`; hang-up/drop.
- **Business rule (FR-C2/C4/C6, D12, ER-A5/C12):** admin sets `available`/`busy`/`unavailable` in
  `admin-users`; only `available` admins are rung. Answer is an **atomic claim** — first admin to
  set `ACTIVE`/`attendedBy` wins; that admin is marked `busy` (skipped for concurrent calls) and
  `MSG_CALL_STOP_RING` is sent to the others. Hang-up → `ENDED` (duration); mid-call drop →
  inactivity timeout. **No recording** (ER-A5). Concurrent callers ring the remaining available
  admins, else voicemail.
- **Affected fields (call-queue):** `status`, `attendedBy`, `answeredOn`, `endedOn`, `durationMs`;
  (admin-users) `availability`.

## 9. Assumptions

- **A1** — Reporters and admins are authenticated internal FrontM users in the same org; this is
  not a public/external portal (REQUIREMENTS §3).
- **A2** — `admin-users` is seeded out-of-band before go-live with at least one PRIMARY and one
  SECONDARY admin; an empty registry triggers the SLA backstop rather than silent loss (ER-B7).
- **A3** — One shared MongoDB cluster; `reports`, `call-queue`, `admin-users` are `shared: true`
  and side-effect-imported by both apps from `lib/collections/`.
- **A4** — Daily.co (VideoCall) + VoIP push are available in the target deployment for calling.
- **A5** — Deployment identifiers (app/bot names, domain, systemId, botIds) are decided at deploy
  time; `deployment.config.json` keeps placeholders until then (D5).
- **A6** — `severity` is initialised from reporter `urgency` (D6) and may be overridden by an admin.

## 10. Constraints

- **C1 — Code-enforced anonymity (non-negotiable).** A single `adminProjection` read gateway
  (`loadReportsForAdmin`/`loadReportForAdmin`); identity-free bot-to-bot payloads and emails;
  reporter ownership asserted on every read/mutate; masked voice calls; audit records admin actions
  only (ER-A2/A3, NFR-1).
- **C2 — Mobile + web with separate per-platform renderers** dispatched once via `state.client`;
  no scattered inline conditionals; shared theme tokens (REQUIREMENTS §9.1, NFR-9).
- **C3 — Custom HTML UI only** for analytics/dashboards — no MongoDB Atlas Charts (D4).
- **C4 — Voice-only, masked, unrecorded** calling (ER-A5, FR-C7).
- **C5 — Routing only through `resolveAssignees`** (D17); no hardcoded role queries.
- **C6 — Optimistic concurrency on every transition** (ER-B5); idempotent jobs (ER-B8).
- **C7 — Server-side validation + sanitisation** of all input before email/HTML use; type+size
  file allow-list (NFR-2, D13).
- **C8 — No unit-test files** (FrontM); verification on the live runtime via `/frontm-review` +
  `/verify` + `npm run build`, with `/lib` helpers kept pure (NFR-7).
- **C9 — Zero duplication** — all shared logic lives once in `/lib` (G3, NFR-6/8).

## 11. Cross-App Message Contract Table (standalone)

> Primary coordination artefact between the two app teams. Identical to §6; reproduced here for
> direct reference. **No payload carries reporter identity.**

| Type | Direction | Payload (identity-free) | Receiver action |
|---|---|---|---|
| `MSG_NEW_REPORT` | user → admin | reportId, category, urgency, severity, assignedTo, createdOn | Notify assigned admins; schedule auto-escalate |
| `MSG_REPORT_REOPENED` | user → admin | reportId, reopenCount, rejectReason | Notify assigned admins |
| `MSG_REPORT_RESOLVED` | admin → user | reportId, resolvedOn | Notify reporter |
| `MSG_REPORT_STATUS_CHANGED` | admin → user | reportId, newStatus | Notify reporter |
| `MSG_REPORT_CLOSED` | admin → user | reportId, closeType | Notify reporter |
| `MSG_INCOMING_CALL` | user → admin | callRef, meetingId (+ VoIP push) | Ring available admins |
| `MSG_CALL_STOP_RING` | admin → admin | callRef, meetingId | Stop ringing the other admins |
| `MSG_ADMIN_NOTIFY` | admin → admin | reportId, event (`NEW`/`REOPENED`/`ESCALATED`) | Recipient-session push-to-self (mobile+web) + email-to-self. Carries the escalate/auto-escalate/manual-log notifications whose acting context is not the recipient's — see framework-mapping rule 32. |

> **Push channel (rule 32).** "push" throughout this BRD means **mobile (APNS/GCM) + web** via
> `sendPushNotifications` to the recipient's own conversation (push-to-self), NOT the web-only
> `sendWebPush`. Reporter dispatches and the X1/X2 admin receivers already run in the recipient's
> session; escalate/auto-escalate/manual-log reach the recipient's session via `MSG_ADMIN_NOTIFY`.
> Email always accompanies push.

  additive later (D17).
