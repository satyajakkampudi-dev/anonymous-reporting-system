# Story Card — Anonymous Reporting System

> LoG.ai Layer 1 (the Synopsis). Source brief: [`../REQUIREMENTS.md`](../REQUIREMENTS.md) +
> [`SPEC.md`](SPEC.md), decisions D1–D17. British English. frontm.ai is always lowercase.
> This is a **2-microapp monorepo**; Layers 1–2 are system-level across both apps.

```
STORY CARD
==========
Problem:       Internal crew and shore staff need a safe channel to report misconduct
               (harassment, safety violations, fraud/ethics, bullying/retaliation) and
               track it to resolution, while a central compliance team triages, resolves
               and escalates reports WITHOUT ever seeing who reported (code-enforced anonymity).

Actors:        - Reporter — authenticated internal crew (any vessel) OR shore-based staff;
                 mobile + web; same org. Creates reports/evidence/calls; reads ONLY own reports.
               - Primary admin (PRIMARY_ADMIN) — central shore-based compliance officer;
                 mobile + web; same org. Triages/resolves/escalates; reads reports identity-stripped.
               - Secondary admin (SECONDARY_ADMIN) — central shore-based senior/escalation
                 (e.g. Designated Person Ashore); everything Primary does PLUS escalated +
                 against-admin reports. Reads reports identity-stripped.
               - System (job scheduler) — non-human actor: auto-escalation, auto-close,
                 SLA-backstop digest timers.
               NOTE (D17): admins are a single CENTRAL shore team, deliberately NOT on the
               reporter's vessel (anonymity). Per-vessel admins are explicitly rejected.
               shipName/location are incident metadata, never routing keys.

Triggers:      - Reporter submits a report (status OPEN; routed PRIMARY, or SECONDARY if againstAdmin).
               - Reporter places an anonymous voice call (rings available admins; voicemail fallback).
               - Reporter accepts / rejects / withdraws (CLOSED_BY_USER / REOPENED once D10 / WITHDRAWN).
               - System auto-escalation timer — CRITICAL +1d, others +3d, if unactioned (D2).
               - System auto-close timer — resolved +30d unacknowledged (D2).
               - System SLA-backstop timer — OPEN 24h / ESCALATED 24h unactioned -> admin digest (D11).
               - Admin logs a manual report (phone/in-person; source=MANUAL).
               - No-answer call -> voicemail -> auto-created report (source=CALL).

Handoffs:      (every cross-app payload is reporter-identity-free)
               - Reporter -> Admin : new report (act) — MSG_NEW_REPORT.
               - Reporter -> Admin : reopen, once (act) — MSG_REPORT_REOPENED.
               - Admin -> Reporter : status changed / resolved / closed (inform; act on RESOLVED) —
                 MSG_REPORT_STATUS_CHANGED / MSG_REPORT_RESOLVED / MSG_REPORT_CLOSED.
               - Admin -> Admin : escalation Primary->Secondary (act) — status->ESCALATED.
               - Reporter -> Admin : incoming anonymous call ring (act) — MSG_INCOMING_CALL (+VoIP push).
               - Admin -> Admin : ring-stop after first answer (inform) — MSG_CALL_STOP_RING.
               Escalation backstop: unactioned ESCALATED raises an SLA digest, never rots (D11).

Micro-apps:    - anonymous-user  — Reporter. mobile + web, own bot. Open to any authenticated user;
                 shows only the caller's own reports. Writes reports + call-queue entries.
               - anonymous-admin — Primary + Secondary admin (role-based visibility within ONE app).
                 mobile + web, own bot, role-gated (license/role + in-code gate via admin-users).
                 Reads reports identity-stripped via adminProjection; runs jobs/analytics/calls.

Cross-app:     Bot-to-bot via state.notification.sendMessageToUserInBot; MSG_* types in lib/constants.js.
               NO payload carries reporter identity.
               - MSG_NEW_REPORT          User  -> Admin : reportId, category, urgency, severity,
                                                           assignedTo, createdOn.
               - MSG_REPORT_REOPENED     User  -> Admin : reportId, reopenCount, rejectReason.
               - MSG_REPORT_RESOLVED     Admin -> User  : reportId, resolvedOn.
               - MSG_REPORT_STATUS_CHANGED Admin -> User: reportId, newStatus.
               - MSG_REPORT_CLOSED       Admin -> User  : reportId, closeType.
               - MSG_INCOMING_CALL       User  -> Admin : callRef, meetingId (+ VoIP push).
               - MSG_CALL_STOP_RING      Admin -> Admin : callRef, meetingId.

Shared data:   One MongoDB cluster, shared: true, suffixed *_${systemId}; both apps side-effect-import
               the definitions from lib/collections/.
               - reports      (reports_${systemId}, audit: true) — the Report entity. Admin reads ONLY
                 via adminProjection (excludes reporterId/contact). Routing via resolveAssignees(report).
               - call-queue   (call_queue_${systemId}) — one entry per call attempt; IDENTITY-FREE.
               - admin-users  (admin_users_${systemId}) — seeded admin registry (D3): adminUserId,
                 adminEmail, role (PRIMARY|SECONDARY), scope (default GLOBAL, D17), availability.
```

## Layer 1 cross-checks (decision-guide validation)

- **Two apps, not one** — Reporter and Admin have *different data-access patterns* (reporter sees
  only own reports; admin sees all, identity-stripped). There is a *handoff* between them and
  *push notifications* across them. The single-vs-multi guide points to **multiple apps**. ✓
- **Two apps, not three+** — Primary and Secondary admin share the same data-access pattern
  (both read via `adminProjection`), differing only by a role filter. One Admin app with
  **role-based visibility** is correct; a separate Secondary app would be over-decomposition. ✓
- **No per-vessel apps** — D17 fixes a *single central compliance team* (`scope = GLOBAL`).
  Vessel is incident metadata, not an app/routing boundary. Future fleet/region scoping is an
  *additive* change inside `resolveAssignees`, not a new app. ✓
- **System (jobs) is an actor, not an app** — auto-escalate/auto-close/SLA timers fire intents
  *inside* anonymous-admin via `state.jobScheduler`; they are triggers, not a separate micro-app. ✓
