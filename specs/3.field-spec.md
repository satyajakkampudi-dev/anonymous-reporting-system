# anonymous-user

## Entity Definition — Report (reporter's view)

```
Entity:       Report
Collection:   reports   (MongoDB reports_${systemId})
Shared:       true
System:       false
NameOverride: false
Architecture: Standalone Collection
              · My Reports list → reportsCollection.loadCollectionWithQuery({ query: { reporterId } })
              · Submit / Detail  → reportDoc.loadDocument({ reportId })  (ownership asserted: reporterId === state.user.userId)
PrimaryKey:   reportId   (RPT- + 10 chars, generated via lib/id-generator.js on first save; retry on unique-index collision — ER-B9)
UserFilter:   reporterId (= state.user.userId; reporter-private, hidden, NOT the primaryKey)
Layout:       Four screens, separate intents (NOT tabs):
              Home/landing (U-F0) · Submit (U-F3/4) · My Reports (U-F1) · Report detail (U-F2)
              + Amend popup (U-F13) · Call compliance (U-F15/16/17)
Architecture decision: TWO-DOC (see §1.5 below)
  Data Doc:    reportDoc          — all fields + sub-collections + persistence. Submit form renders this (standard sections). Never sent for display screens.
  Display Doc: reportDisplayDoc   — CardsSet sections for Home, My Reports list, Detail. sendResponse() renders this.
```

### Infrastructure fields (derived — NOT PM-specified, NON-NEGOTIABLE)

These never appear in a wireframe; they are framework requirements of the load/query pattern.

| Field | Type | Properties | Source / purpose |
|---|---|---|---|
| `reportId` | text | `primaryKey: true`, `hidden: true` | Generated `state.getUniqueId()`-style via `lib/id-generator.js` (`RPT-`+10). Unique per row. Without it, `save()` upserts `{}` and the collection dedupes all rows to one. |
| `reporterId` | text | `hidden: true`, `dbName: "reporterId"`, NOT primaryKey | `state.user.userId`. Sets the `loadCollectionWithQuery` filter and the ownership assertion. Reporter-private — excluded from `adminProjection`. |
| `status` | text | `hidden: true` (on submit) | Set `OPEN` at submit (U-F8). Read-only displayed on detail. |
| `severity` | text | `hidden: true` | Derived from `urgency` at submit (D6); admin may override. Not editable by reporter. |
| `source` | text | `hidden: true` | `REPORTER` (submit) / `CALL` (voicemail) / `MANUAL` (admin). |
| `assignedTo` | text | `hidden: true` | `resolveAssignees(report)` at submit (D17). |
| `createdOn` / `updatedOn` | number | `hidden: true` | ms timestamps. `updatedOn` backs optimistic-concurrency (ER-B5). |
| `version` | number | `hidden: true` | Incremented per write; transition guard. |
| `reopenCount` | number | `hidden: true` | Default 0; 0→1 once (D10). |
| `withdrawnOn` / `resolvedOn` | number | `hidden: true` | Set on the respective terminal/resolved transitions. |

> **autoSave bootstrap (CRITICAL).** `reportDoc` is `autoSave: true` (U-F9 draft autosave). Therefore
> `src/main.js` MUST set `state.onConfig = () => { state.contextAware = true; }` AND `main.onResolution`
> MUST call `await Context.CreateAndInit("mainApp", { state })` before any `loadDocument`. See BRD §8.1
> and CLAUDE.md "App Entry-Point Bootstrap". Subsequent popup intents (Amend) use `Context.Create`, never re-`loadDocument`.

## §1.5 Rendering Architecture Decision — anonymous-user

**Two-Doc.** §9 of REQUIREMENTS specifies custom HTML cards ("sailors-cart style") for Home, My
Reports, and Detail; the Submit form is a standard Form Doc. Any `custom_card`/`display_only`
section forces Two-Doc.

```
Architecture: Two-Doc
Data Doc:     reportDoc           (always present — owns every Field + both sub-collections)
Display Doc:  reportDisplayDoc    (Home landing card, My Reports list card, Detail cards)
Sections needing Display Doc: Home landing, My Reports list, Report header, Reporter content (detail),
                              Status timeline, Resolution, Amendment list (detail view)
```

Per §9.1, every Display section dispatches once via `renderForPlatform(data, { web, mobile })` —
distinct `web.js`/`mobile.js` renderers, shared theme tokens (`lib/utils/theme.js`).

The submission form (U-F3/F4). Standard Form Doc fields on `reportDoc`. `columns: 2` where noted;

| `urgency` | dropdown | ✅ | ✅ | 1 | Options: Immediate risk · High · Medium · Low. Drives initial `severity` (D6). |

free text). `description`→TEXT_AREA (multi-line, not single-line TEXT). All confirmed.

## Section: Contact (Submit form) — rendering: `standard`

Reporter-private contact channel (U-F7). Conditional validation per method. **Reporter-private —
excluded from `adminProjection`; `contactValue` is `encrypted: true`.**

| Field | Type | Mandatory | Editable | Column | Notes |
|---|---|:--:|:--:|:--:|---|
| `contactMethod` | dropdown | ✅ | ✅ | 0 | Options: None · Email · Phone · Cabin number. Default None. |
| `contactValue` | text | conditional | ✅ | 1 | `encrypted: true`. Hidden/cleared when method = None. Validated per method (U-F7): Email→RFC-pragmatic regex; Phone→E.164-tolerant; Cabin→alphanumeric. Mandatory only when method ≠ None. |

> **Conditional reveal.** `contactValue` is shown/required only when `contactMethod ≠ None`. The
> framework has no declarative "show-if"; this is enforced in `contactValue.onValidation` +
> `contactMethod.onSave` (validation by current method) — a custom frame (U-F7).
> **Field-type note:** `contactValue` stays a single TEXT field (not EMAIL/PHONE) because its meaning
> changes with `contactMethod`; the per-method validator lives in `lib/validation.js`.

## Cross-app contracts — anonymous-user side

```yaml
# SENDS
- id: U-X1
  message_type: MSG_NEW_REPORT
  to_bot: anonymous-admin
  trigger: after reportDoc.save() in U-F8 (submit) and U-F16 (voicemail auto-create)
  payload: [reportId, category, urgency, severity, assignedTo, createdOn]   # identity-free
- id: U-X2
  message_type: MSG_REPORT_REOPENED
  to_bot: anonymous-admin
  trigger: after U-F11 reject persists
  payload: [reportId, reopenCount, rejectReason]
- id: U-X3
  message_type: MSG_INCOMING_CALL
  to_bot: anonymous-admin
  trigger: U-F15 after RINGING entry created
  payload: [callRef, meetingId]   # + VoIP push

# RECEIVES (onMatching, Context B — load report by reportId before reading)
- message_type: MSG_REPORT_RESOLVED      # from anonymous-admin → notify reporter (U-X4/U-F14)
- message_type: MSG_REPORT_STATUS_CHANGED # (U-X5/U-F14)
- message_type: MSG_REPORT_CLOSED         # (U-X6/U-F14)
```

> **Send after save** (never before persistence). **Minimal payload** — IDs + display text only;
> the receiver loads full data from the shared `reports` collection through `adminProjection`.
> Incoming messages run in **sandbox** — persist any state via `setPersistedField`.

# anonymous-admin

> Layer 3 pass for the admin console. Three entities over the **shared** collections (`reports`,
> `admin-users`, `call-queue`), all read through the single anonymity gateway. **No admin Doc binds
> any reporter-identity field** (`reporterId`, `contactMethod`, `contactValue`, reporter-create audit)
> — the `adminProjection` (SPEC.md) is enforced at both the query layer (`loadReportsForAdmin` /
> `loadReportForAdmin`, ER-A3) and the Doc-binding layer (this spec).
>
> **Decisions taken this pass:** D-L3-2 `adminReportDoc` is `autoSave: true` + Context bootstrap
> (BRD §8.2). D-L3-3 small-cell suppression threshold = **fewer than 5** (k-anonymity k=5, ER-A6).
> D-L3-4 manage transitions use **per-action `sendQuickFormResponse()` popups** (A-F8–F12). D-L3-5
> manual log carries **report content + evidence only** (no contact channel — `source = MANUAL`,
> `reporterId` empty).

## Entity Definition — Report (admin's view)

```
Entity:       Report (admin view — adminProjection field set only)
Collection:   reports   (MongoDB reports_${systemId})   ← SAME shared collection as anonymous-user
Shared:       true
System:       false
NameOverride: false
Architecture: Standalone Collection, read ONLY through the anonymity gateway:
              · Queue      → loadReportsForAdmin({ query, projection: adminProjection })  (lib/access.js, ER-A3)
              · Manage     → loadReportForAdmin({ reportId })   (projection applied; NO ownership filter — role-gated, not owner-gated)
              · Manual log → adminReportDoc fresh; reportId generated on first save (lib/id-generator.js)
PrimaryKey:   reportId   (shared PK; admin never generates one except on manual log)
UserFilter:   NONE by reporterId. The admin queue is filtered by ROLE + recusal (A-F4), never by owner.
Layout:       Separate screens via separate intents (NOT tabs):
              Dashboard (A-F2) · Queue (A-F3/4/5) · Manage detail (A-F6…F12) · Manual log (A-F13)
              · Alerts/Digest (A-F19) · On-call (A-F20) · Incoming call (A-F21/22)
Architecture decision: TWO-DOC (custom HTML throughout, REQUIREMENTS §9)
  Data Doc:    adminReportDoc      — adminProjection fields + admin-entered fields + statusHistory/amendments (read-only view) + persistence. autoSave: true. NEVER sent for display.
  Display Doc: adminDisplayDoc     — every CardsSet section (Dashboard, Queue, Manage cards, Alerts, On-call, Incoming call). sendResponse() renders this.
  Aux Data Docs: adminUserDoc (admin-users — on-call availability) · callQueueDoc (call-queue — answer/claim/end)
```

> **No identity binding (C1, ER-A2/A3).** `adminReportDoc` declares **no** `reporterId`,
> `contactMethod`, or `contactValue` field, and no reporter-create audit field. Every read flows
> through `loadReportsForAdmin` / `loadReportForAdmin` which apply `{ projection: adminProjection }`
> — no admin code path queries `reports` directly. Bot-to-bot payloads and admin emails/HTML are
> built from this set only.

## §1.5 Rendering Architecture Decision — anonymous-admin

**Two-Doc.** REQUIREMENTS §9 mandates custom HTML (stat cards, queue cards, manage cards, no Atlas
Charts — D4); §9.1 mandates separate per-platform renderers. Any `custom_card`/`display_only` section
forces Two-Doc.

```
Architecture: Two-Doc
Data Docs:    adminReportDoc (reports, autoSave) · adminUserDoc (admin-users) · callQueueDoc (call-queue)
Display Doc:  adminDisplayDoc
Sections needing Display Doc: Dashboard, Report queue, Manage header, Manage reporter-content,
                              Manage status-timeline, Manage amendments (read-only), Manage resolution,
                              Manage actions, Alerts/Digest, On-call card, Incoming-call card
Standard (inline) sections on a Data Doc: Manual log form + the per-action popup sections (resolve,
                              escalate, severity override, close-rejected) — all on adminReportDoc.
```

Per §9.1, every Display section dispatches once via `renderForPlatform(data, { web, mobile })`.
